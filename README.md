# Muzz Chat ‚Äì Android Exercise

A chat interface built for the Muzz Android recruitment exercise.

---

## Architecture

### Modularization

The project consists of two modules:

- **`:app`** ‚Äì entry point, Koin setup, MainActivity
- **`:feature-chat`** ‚Äì self-contained chat feature with a single public entry point

Within `:feature-chat`, Clean Architecture layer separation is enforced through packages:


All classes are `internal` ‚Äî only `ChatScreen` (Composable) and `chatModule` (Koin) are public.

At larger scale I would split the feature into sub-modules per layer (`:feature-chat:data`,
`:feature-chat:domain`, `:feature-chat:presentation`) and extract a `:core` modules for shared
infrastructure such as `UserRepository`, `DateFormatter`, and the app theme.

### MVI

The presentation layer follows MVI:

- `ChatViewModel` exposes a single `StateFlow<ChatState>` and accepts `ChatIntent` via
  `processIntent()`
- All state mutations go through `ChatReducer` ‚Äî an object of pure extension functions on
  `ChatState`
- This makes state transitions predictable and the reducer trivially unit-testable

### Use Cases

Use Cases are only created where they encapsulate real business logic. The ViewModel interacts with
repositories directly for simple reads. The only Use Case is `SendMessageUseCase` which handles:

- Sending the user's message
- Triggering the conflated auto-reply via an internal `MutableSharedFlow`

## Data Layer

### Room + Flow

Messages are persisted in a Room database. The DAO returns `Flow<List<MessageEntity>>` which emits
the full updated list on every INSERT ‚Äî satisfying the observable architecture requirement.

### Message Model

`Message` carries an `isOutgoing: Boolean` field. This reflects how a real server API would return
messages already contextualised to the authenticated user. The local mock simulates this behaviour.

### User Data & Avatar

Only the other user (Sarah) is stored in `UserRepositoryImpl`. The current user's identity is
implicit in `isOutgoing` on each message.

The avatar is stored as a drawable resource name string (e.g. `"avatar_sarah"`) in the data layer
and resolved to a resource ID at runtime in the Composable via `getIdentifier()`. This simulates how
a real API would return an image URL ‚Äî the data layer provides a string reference, the UI resolves
it ‚Äî while avoiding any external network dependency.

### Timestamps

Timestamp generation is handled by a `TimeProvider` interface injected into `SendMessageUseCase`.
The interface also makes the Use Case fully testable with a deterministic fake.

## Auto Reply

Initially, auto-reply was implemented using `debounce(5000ms)`. The problem was that with such a
short delay, the gap between the user's messages and the auto-reply was always under 20 seconds,
causing them to be always visually grouped ‚Äî the ungrouped appearance between messages was never
visible.

To properly demonstrate grouping behaviour, the delay was increased to 30 seconds (above the
grouping
threshold). Instead of `debounce`, `conflate` + `delay(30s)` was used. `conflate` processes the
first
emission immediately and keeps the last one received while the downstream is busy ‚Äî discarding
everything
in between. This means Sarah replies to the first message after 30 seconds, then replies again to
the
last message sent during that delay. Once that second reply is sent, the next burst is processed
independently.

With `debounce` the timer would reset on every message, requiring 30 seconds of silence before any
reply.
With `conflate` the timer starts from the first message in a burst, reducing wait time during
testing
and making the conversation feel more natural.

This has an interesting side effect: Sarah can send two messages in a row ‚Äî one response for the
first
message in the burst and a second response for the last.

This was a last-minute change introduced as the simplest solution. The reply text is generated by
the
Repository (`generateMessage()`), keeping the response strings as a data concern replaceable by a
real API.

## UI

- Built entirely with **Jetpack Compose**
- `Scaffold` with `TopBar` (chat header) and `BottomBar` (message input)
- `LazyColumn` with `reverseLayout = true` ‚Äî newest messages at the bottom
- Message bubbles use `RoundedCornerShape` with one flattened corner to indicate direction
- Colors use Material3 semantic tokens: `primary/onPrimary` for outgoing,
  `surfaceVariant/onSurfaceVariant` for incoming

### Message Rules

| Rule        | Behaviour                                                                        |
|-------------|----------------------------------------------------------------------------------|
| Grouping    | Messages from the same user within 20 seconds get reduced spacing                |
| Date header | Inserted when gap from previous message exceeds 1 hour, or at conversation start |
| Format      | `{day} {HH:mm}` e.g. `Thursday 11:59`                                            |

## Testing

Tests cover:

- **`ChatReducer`** ‚Äì grouping logic, date header insertion, input state transitions, message
  ordering
- **`SendMessageUseCase`** ‚Äì message content and timestamp, conflate,
  timing
- **`ChatViewModel`** ‚Äì orchestration, state updates, intent handling

Compose UI and Repository tests were omitted due to time restrictions. Given more time I would also
add DAO integration tests.

## Assumptions

- Only the other user is stored in `UserRepository` ‚Äî current user identity is encoded in
  `isOutgoing`
- `isOutgoing` mirrors how a real server API contextualises messages for the authenticated user
- No network connectivity ‚Äî all messages created and stored locally as required
- No login screen ‚Äî app launches directly into the chat between two fixed users
- Auto-reply delay of 5 seconds represents a natural conversation pause
- Colors approximated from exercise screenshots
